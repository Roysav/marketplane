# Introduction

**A control plane for trading systems.**

Define custom record types declaratively, persist them, and watch for real-time changes. Built for trading across crypto, prediction markets, and traditional finance.

## Purpose

Make the process of building an automated trading program easier, safer, and enable fast development cycles. Allow dynamic auto-portioning for rolling new / updated strategies, scale their quota slowly.

## Reasoning

When building a trading system, specifically algo-trading, there are two centric parts:
1. Modeling - Answers the question "What's the fair value of asset X?"
2. Execution - How do we handle volatility, fees and market making incentives.


### Overlaps

While it may be easy to dismiss both **modeling** and **execution** as two different parts, they sometime overlap.

The first overlap is **certainty**, execution might depend on how confident a specific model is about his evaluation.
We may want a **market-maker** to change the spread based on that confidence score alongside the IV of the asset.

To try and fix this overlap, and where each of those sits, I made some early mental model:

### Modeling

In addition to just pricing the asset, it should produce a confidence field, and choose some execution strategy declaratively.
A modeling agent should tell us:
- What is the fair value of underlying
- How much (in percentages) does it want to expose is allowance to this trade.
- How much spread does it need for (1% / 1 cent need to be decided in the future, or be case-by-case)
- What execution strategy the execution controller should use. For example, **AggressiveBookKeeper** might watch the order book, and try and be the first one, up until the fair value discounted by the spread. The modeling part should of course include any additional parameters.


## Features

- **Extensible Record System** — Define custom types via MetaRecord, StreamDefinition
- **Dual Storage Model** — Row (persistent) and Stream (real-time) storage
- **Tradespace Isolation** — Namespace-like boundaries for resources
- **Schema Validation** — JSON Schema validation for record specs
- **Ledger & Allocations** — Append-only fund ledger with atomic balance checks

## Concepts & Core Entities

All records are identified by a type string: `group/version/kind`

| Type | Scope | Storage | Description |
|------|-------|---------|-------------|
| `core/v1/Tradespace` | global | Row | Isolation boundary for records (like a namespace). All tradespace-scoped records belong to one. |
| `core/v1/MetaRecord` | global | Row | Defines a custom Row-storage record type along with its JSON Schema. |
| `core/v1/StreamDefinition` | global | Row | Defines a custom timeseries stream type, its schema, and retention policy. |
| `core/v1/Quota` | tradespace | Row | Sets starting balance limits per tradespace per currency. |
| `core/v1/Allocation` | tradespace | Row | Requests a fund allocation from a tradespace balance. Goes through Pending → Approved/Rejected lifecycle. |

Custom types such as `polymarket/v1/Order` are registered by creating a `MetaRecord`.

### Record Structure

Every record shares the same envelope:

```json
{
  "group": "polymarket",
  "version": "v1",
  "kind": "Order",
  "metadata": {
    "name": "order-001",
    "tradespace": "my-ts",
    "labels": {"strategy": "mean-revert"},
    "resourceVersion": 1
  },
  "spec": {
    "marketId": "abc",
    "side": "YES",
    "amount": "10.00"
  },
  "status": {}
}
```

- **TypeMeta** (`group/version/kind`) — what type the record is.
- **ObjectMeta** (`name`, `tradespace`, `labels`, `resourceVersion`) — identity and metadata.
- **Spec** — desired state, validated against the type's JSON Schema.
- **Status** — current state, written by controllers.

### Tradespace

A tradespace is an isolation boundary. Records scoped to `tradespace` must carry a non-default tradespace name. Records scoped to `global` (like `core/v1/MetaRecord`) always live in the `"default"` tradespace.

### Schema Validation

When creating or updating a record, the service validates `spec` against:
1. A hardcoded schema for core types (e.g., `core/v1/Quota` requires `balances`).
2. The `schema` field of the matching `core/v1/MetaRecord` for custom types.

Validation uses [gojsonschema](https://github.com/xeipuuv/gojsonschema).

## Quick Walkthrough

### 1. Define a tradespace

```json
{
  "group": "core", "version": "v1", "kind": "Tradespace",
  "metadata": {"name": "algo-ts"},
  "spec": {"description": "My algo trading tradespace"}
}
```

### 2. Register a custom record type

```json
{
  "group": "core", "version": "v1", "kind": "MetaRecord",
  "metadata": {"name": "Order.polymarket"},
  "spec": {
    "group": "polymarket", "version": "v1", "kind": "Order",
    "scope": "tradespace",
    "schema": {
      "type": "object",
      "required": ["marketId", "side"],
      "properties": {
        "marketId": {"type": "string"},
        "side": {"type": "string", "enum": ["YES", "NO"]}
      }
    }
  }
}
```

### 3. Create a record of the custom type

```json
{
  "group": "polymarket", "version": "v1", "kind": "Order",
  "metadata": {"name": "order-001", "tradespace": "algo-ts"},
  "spec": {"marketId": "market-abc", "side": "YES"}
}
```

### 4. Watch for changes

Call `RecordService.Watch` with `type = "polymarket/v1/Order"` to receive a server-sent stream of `created`, `updated`, and `deleted` events.
